# AI智建 -- 面试问答准备

> 适用场景：毕业答辩、导师面试、企业技术面试、竞赛路演
> 项目：基于 pyRevit 的自然语言驱动智能建模系统

---

## 一、项目概述类

### Q1: 请简单介绍一下你的项目？

**A:**

- AI智建是一个 Revit 插件，核心目标是**用自然语言代替手动操作来完成结构建模**。它有两种工作模式：参数化一键生成框架结构，以及通过中文对话驱动 AI 建模。
- 技术上，用户用中文说出建模需求，系统调用 DeepSeek 大模型将自然语言解析为结构化 JSON 指令，再由统一的建模引擎通过 Revit API 执行——整个过程从输入到出模型，几秒钟完成。
- 目前系统支持 10 种动作类型（柱、梁、板的创建，框架生成，截面修改，构件删除，数量查询，明细查询，统计汇总等），配套 289 个离线测试用例和 37 个回归测试场景。

> **加分回答：** 如果面试官追问规模，可以补充："插件包含 8 个功能按钮，分布在 5 个面板中；AI 对话模块支持多轮上下文、失败自动恢复、指令回放等生产级特性。"

---

### Q2: 为什么选择这个课题？

**A:**

- **行业痛点真实存在。** 传统 Revit 建模门槛高，操作繁琐，一栋教学楼的框架结构手动建模可能要一两天，而且对操作者的软件熟练度有很高要求。
- **智能建造是学科方向。** 我本身是智能建造专业的学生，"AI + BIM"正好是学科前沿和行业趋势的交汇点，选这个题目既能深入专业，又能实践 AI 工程化能力。
- **个人动机。** 我在学习 Revit 的过程中亲身体会到重复建模的痛苦，就想："能不能直接告诉电脑我要什么，让它自己画？"这个想法驱动了整个项目。

> **加分回答：** "住建部在推 BIM 正向设计，但一线设计师的建模效率是落地瓶颈。如果能降低 BIM 的使用门槛，对行业数字化转型有实际推动作用。"

---

### Q3: 这个项目解决了什么实际问题？

**A:**

- **效率问题。** 一栋 5 层框架结构，参数化模式下输入跨数、层高、截面，几秒钟生成完整的轴网、标高、柱、梁、板，手动建模可能要一整天。
- **门槛问题。** Revit 的学习曲线很陡，尤其对非计算机背景的土木/建筑专业学生。AI 模式下用户直接说"在第二层加一根 300x700 的梁"，不需要知道 Revit 的菜单在哪里。
- **重复劳动问题。** 框架结构里大量柱梁板是规律性排布的，参数化批量生成比逐个手动放置效率高几个数量级。
- **数据衔接问题。** 支持 Excel 导入结构参数，设计院常用的"结构计算表 -> BIM 模型"的数据链路被打通了。

## 二、技术架构类

### Q4: 系统的整体架构是怎样的？

**A:**

- 系统分四层：**表现层**（pyRevit 按钮 + WPF 窗口）、**解析层**（DeepSeek API + JSON 解析器）、**引擎层**（create_column、create_beam 等核心函数）、**平台层**（Revit API）。
- 最关键的设计决策是**两种模式共享同一套建模引擎**。参数化面板直接调用引擎函数，AI 对话模式经过 LLM 解析后生成的 JSON 指令也路由到同一套函数。这意味着引擎只需要维护一份代码，两个入口天然保持一致。
- 解析层的职责被严格限定：接收自然语言，输出结构化 JSON。引擎层只接受结构化参数，不关心输入来源。这种解耦让离线测试成为可能——引擎函数不依赖 Revit，可以在 Mac 上用 pytest 跑。

> **加分回答：** "这个架构也为未来扩展留了空间。比如后续要加移动端远程控制，只需要在表现层加一个 WebSocket 入口，解析层和引擎层完全复用。"

---

### Q5: 为什么选择 pyRevit 而不是 Revit Add-in (C#)？

**A:**

- **开发效率。** Python 的开发速度比 C# 快很多，特别是在原型阶段。pyRevit 提供了热加载，改完代码不用重启 Revit 就能生效，大幅缩短调试周期。
- **生态优势。** Python 有丰富的数据处理库（json、openpyxl），API 调用也比 C# 简洁。做 AI 集成时，Python 处理 JSON 和 HTTP 请求非常自然。
- **WPF 支持。** pyRevit 内置了 WPF 窗口支持，可以用 Python 直接定义界面，不需要额外的 XAML 设计器。
- **代价是 IronPython 兼容性。** pyRevit 运行在 IronPython 2.7 上，不完全兼容 CPython。比如所有中文字符串要加 `u""` 前缀，`basestring` 类型检查不能省略。这些坑我在开发中逐一解决了。

> **加分回答：** "如果项目未来要做高性能计算（比如实时结构分析），可能会考虑 C# 插件。但当前阶段，快速迭代和 AI 集成的优先级更高，Python 是更合理的选择。"

---

### Q6: 为什么选择 DeepSeek 而不是 GPT/Claude？

**A:**

- **中文能力。** 这个项目的用户输入全部是中文，包括"首层""二层柱""300x700 梁"这类建筑领域的中文表达。DeepSeek 在中文理解上表现非常好，不需要额外的翻译层。
- **成本。** DeepSeek 提供免费额度，对学生项目非常友好。GPT-4 的 API 成本高出几个数量级。
- **稳定性。** DeepSeek API 在国内网络环境下延迟低、可用性高，不需要翻墙或代理。
- **可替换设计。** 系统的配置模块支持三级优先级（环境变量 > config.json > 默认值），API 地址、密钥、模型名都可以通过配置切换。如果未来要换成 GPT、Claude 或其他兼容 OpenAI 格式的模型，改一下配置文件就行，代码不用动。

> **加分回答：** "我在 config.py 里用了通用别名机制，`LLM_API_URL`、`LLM_API_KEY`、`LLM_MODEL` 这些别名覆盖了 DeepSeek 专属配置名，体现了对供应商锁定的预防。"

---

### Q7: 如何保证大模型输出的稳定性？

**A:**

- **严格的 system prompt。** prompt 明确规定"只能输出合法 JSON，不要任何解释文字"，并设置 `temperature=0.1` 和 `response_format: json_object`，从源头压缩随机性。
- **16 个 few-shot 示例。** prompt 里内置了 16 个覆盖所有 action 类型的示例，包括边界情况（省略参数、中文楼层表达、单数字截面补全），让模型有明确的模仿对象。
- **多层容错。** 解析器能处理 markdown 代码块包裹的 JSON（模型有时会加 ` ```json ` 标记）；如果 JSON 里的 action 是中文别名（如"创建柱子"），解析器会自动映射到标准 action 名。
- **回归测试。** 维护了 37 个从真实 AI 回复中提取的测试用例，覆盖各种非标准输出格式。每次修改解析逻辑后都跑一遍，确保不会引入退化。
- **重试机制。** API 调用失败时自动重试（指数退避），最多重试 2 次；5xx 错误和网络异常触发重试，4xx 错误直接报错。

> **加分回答：** "稳定性不是靠单一手段保证的，而是从 prompt 约束、输出解析、参数校验、执行验证四个层面叠加防御。任何一层没拦住的问题，下一层都有机会捕获。"

---

### Q8: 如何处理大模型的"幻觉"问题？

**A:**

- **输出格式校验。** 模型返回的 JSON 必须包含 `action` 和 `params` 字段，action 必须在支持的 10 种类型中。不符合格式的直接拒绝并提示用户重新输入。
- **参数范围校验。** 截面尺寸有上下限（100mm~2000mm），层高有合理范围（2000mm~10000mm），楼层编号会和当前模型实际标高做比对。超出范围的参数在引擎层被拦截，给出具体的中文错误提示。
- **Revit API 兜底。** 即使前面的校验都没拦住，Revit API 本身会拒绝不合理的操作（比如在不存在的标高上放构件），这是最后一道防线。
- **用户友好的错误恢复。** 每种错误类型都有专门的恢复建议。比如"楼层超出范围"会提示"请按当前模型实际楼层范围重试"，"缺少参数"会列出具体缺哪些字段。

> **加分回答：** "幻觉问题不可能完全消除，但可以通过缩小输出空间来降低影响。我的 prompt 把模型的输出严格限定在一个 JSON schema 里，相比让模型自由生成代码，幻觉的危害面小很多。"

## 三、核心算法类

### Q9: 参数化框架生成的算法是什么？

**A:**

- 算法核心是**坐标推导 + 嵌套循环**：
  1. 根据 X 方向跨距列表累加得到 X 轴坐标序列，Y 方向同理。比如 `x_spans=[6000, 6000, 6000]` 得到 `x_coords=[0, 6000, 12000, 18000]`。
  2. 生成轴网：X 坐标生成竖向轴线，Y 坐标生成横向轴线。
  3. 生成标高：根据层数和层高（支持首层特殊层高）创建标高序列。
  4. 逐层生成构件：对每一层，X 坐标 x Y 坐标的交叉点放柱子；沿轴线方向放梁（X 向梁和 Y 向梁的截面可以不同）；每个封闭矩形区域铺楼板。
- **坐标转换。** 用户输入毫米，Revit 内部用英尺，有一个统一的 `mm_to_feet` 转换函数（1 英尺 = 304.8 毫米）。
- 整个生成过程在一个 Revit Transaction 内完成，要么全部成功，要么全部回滚，保证模型一致性。

> **加分回答：** "这个算法的时间复杂度是 O(F x (Nx x Ny))，F 是层数，Nx/Ny 是 X/Y 方向网格数。对于一个 5 层 4x3 框架，总共生成 5x20=100 根柱、5x31=155 根梁、5x12=60 块板，整个过程在 Revit 里不到 10 秒。"

---

### Q10: AI 指令的解析流程是怎样的？

**A:**

- 完整流程：**用户中文输入 -> 拼装对话历史 -> 调用 DeepSeek API -> 提取 JSON -> 动作分发 -> 引擎执行 -> 结果反馈**。
- 具体步骤：
  1. 用户输入和对话历史一起发送给 DeepSeek（system prompt + 最近 20 轮对话）。
  2. API 返回的文本经过 `parse_command()` 处理：先尝试 `json.loads()`，失败则用正则提取 markdown 代码块中的 JSON。
  3. 提取出的 `action` 经过别名映射（比如 `"创建柱子"` 映射到 `"create_column"`，`"add_beam"` 映射到 `"create_beam"`），别名表有 40+ 条。
  4. 映射后的标准 action 通过 dispatcher 路由到对应的引擎函数。
  5. 引擎函数校验参数、调用 Revit API 创建构件，返回执行结果。

> **加分回答：** "解析器对批量指令也有处理——如果 AI 返回的是 JSON 数组（用户说'在 A1 和 B1 各放一根柱子'），会逐条派发执行并汇总结果。"

---

### Q11: 多轮对话是如何实现的？

**A:**

- 维护一个 `conversation` 列表，每个元素是 `{"role": "user"/"assistant"/"system", "content": "..."}` 格式的字典。
- 每次 API 调用都把完整的对话历史发送过去，让模型理解上下文。比如用户先说"建一个 3x2 框架"，再说"把柱子改成 600"，模型能理解"柱子"指的是刚才建的框架里的柱子。
- **自动裁剪。** 对话超过 20 轮（可配置）时，自动丢弃最早的轮次，只保留最近的历史。裁剪时确保保留的第一条消息是 user 角色（不会以 assistant 开头导致上下文断裂）。
- 用户可以用 `/reset` 命令手动清空对话历史，开始新的对话上下文。

> **加分回答：** "裁剪策略选择了简单的滑动窗口而不是 token 计数，原因是 DeepSeek API 的 token 限制对当前场景足够宽裕，滑动窗口的实现更可靠、更容易测试。"

## 四、工程实践类

### Q12: 项目的测试策略是什么？

**A:**

- **289 个离线测试，零 Revit 依赖。** 所有测试在 Mac 上用 pytest 运行，不需要安装 Revit。这是通过依赖注入实现的——引擎函数接受 `doc` 和 `DB` 作为参数而不是直接 import，测试时传入 mock 对象。
- **测试覆盖面。** 20 个测试文件覆盖：配置加载、prompt 内容、JSON 解析、别名映射、参数校验、坐标转换、楼层映射、回归用例、Excel 导入、模型导出、日志记录等。
- **回归测试。** 维护了 37 个从真实 AI 回复中提取的测试用例（`ai_reply_regression_cases.json`），覆盖各种非标准输出。每次改解析逻辑后跑一遍。
- **开发约束。** 每个新功能必须有对应的测试，新 action 必须在 6 个位置同步更新（别名、恢复、日志、prompt、回归用例、分发测试），这是写在 CLAUDE.md 里的硬性规定。

> **加分回答：** "测试不只是验证功能正确性，也是文档。新人看测试用例就能理解每个函数的输入输出契约，比注释更可靠。"

---

### Q13: 跨平台开发是怎么做的？

**A:**

- **Mac 开发 + Windows 运行。** 代码编写和测试在 Mac 上完成（VSCode + pytest），Revit 实测在 Windows 机器上进行。代码通过 Git 同步。
- **IronPython 兼容。** pyRevit 运行环境是 IronPython 2.7，和标准 CPython 有差异。主要踩过的坑：
  - 所有中文字符串必须加 `u""` 前缀，否则在 IronPython 下会乱码。
  - `basestring` 类型在 CPython 3 里不存在，需要 `try/except` 兼容。
  - `.NET` 的 `HttpWebRequest` 替代了 Python 的 `requests` 库（IronPython 不支持）。
- **conftest.py 统一处理。** 测试的 `conftest.py` 把模块搜索路径配好，让 Mac 上的 pytest 能正确 import `AISmartBuild.extension/lib/` 下的模块。

---

### Q14: 如何管理项目配置？

**A:**

- **三级优先级：** 环境变量 > `~/.ai-smart-build/config.json` > 代码默认值。这意味着部署时可以用环境变量覆盖配置，开发时用配置文件，什么都不设就用合理的默认值。
- **通用别名。** 每个配置项都有多个别名，比如 `DEEPSEEK_API_KEY` 也可以用 `LLM_API_KEY`。这样切换 API 提供商时，配置键名不会成为障碍。
- **启动校验。** `validate_config()` 在启动时检查 API Key 是否为空或占位符、URL 是否合法、模型名是否设置，有问题就给出中文警告。
- **设置面板。** 插件内有一个 Settings 按钮，点开是 WPF 窗口，可以在 Revit 界面里直接修改配置，不用手动编辑 JSON 文件。

---

### Q15: 遇到过最大的技术困难是什么？

**A:**

- **困难一：IronPython 兼容性。** 刚开始在 Mac 上写好的代码放到 Revit 里全是编码错误。花了不少时间才搞清楚是 IronPython 对 unicode 处理的差异。解决方案是给所有中文字符串加 `u""` 前缀，并在 CI 规范里写死这条规则。
- **困难二：LLM 输出不稳定。** 早期版本的 AI 回复经常夹带解释文字、用 markdown 包裹 JSON、甚至返回中文 action 名。解决方案是：加强 prompt 约束、增加 few-shot 示例、写容错解析器（能从 markdown 块里提取 JSON、能映射中文 action 别名）、建立回归测试防退化。
- **困难三：离线测试架构设计。** Revit API 只能在 Revit 进程内调用，但我需要在 Mac 上跑测试。解决方案是依赖注入——引擎函数不直接 import `DB`，而是通过参数传入。测试时传 mock 对象，运行时传真实的 Revit DB。这个设计决策让 289 个测试全部能离线运行。

> **加分回答：** "每个困难的解决都不是一次性的。比如 LLM 稳定性，我迭代了 prompt 三四个版本，每次都是发现新的失败模式、加测试用例、调整 prompt、再验证。这个过程让我理解了什么叫'工程化地使用 AI'。"

## 五、创新与展望类

### Q16: 项目的创新点在哪里？

**A:**

- **LLM x BIM 融合。** 据我调研，目前市面上还没有成熟的中文自然语言驱动 BIM 建模产品。学术论文里有类似探索，但大多停留在概念阶段，我的项目有完整的工程实现。
- **双模统一架构。** 参数化模式和 AI 模式共享建模引擎，这个设计在同类项目中不常见。大多数项目要么只做参数化，要么只做 AI，两条路互不相通。
- **全链路容错。** 从 prompt 约束到 JSON 解析到参数校验到执行恢复，建立了完整的错误处理链路，不是"能跑就行"的 demo 级别。
- **低门槛交互。** 用户不需要学 Revit 操作，不需要写代码，用自然语言就能完成建模。对建筑学、土木工程等非软件背景的学生和从业者来说，大幅降低了 BIM 使用门槛。

---

### Q17: 和市面上的 BIM 插件相比，你的优势是什么？

**A:**

- **自然语言界面。** 现有的 Revit 参数化插件（如 Dynamo）需要学习可视化编程，仍有学习曲线。市面上没有产品能做到"用中文说一句话就建模型"。
- **开放架构。** 配置系统支持切换 LLM 提供商（DeepSeek、GPT、本地模型都可以），不绑定单一供应商。
- **参数化 + AI 一体化。** 规则性强的用参数化批量生成，灵活调整用 AI 对话。两种模式在同一个插件里无缝衔接，而不是两个独立工具。
- **可测试性。** 289 个离线测试是工程质量的保证。很多开源 Revit 插件连一个测试都没有。

---

### Q18: 后续有什么改进计划？

**A:**

- **规范校验集成。** 接入 GB 50010 等结构设计规范，自动检查截面尺寸、配筋率、轴压比等是否满足规范要求。
- **更多结构体系。** 当前只支持框架结构，后续计划支持框架-剪力墙、框架-核心筒等复杂体系。
- **配筋设计。** 根据结构分析结果自动生成钢筋模型，形成"建模 -> 分析 -> 配筋"的闭环。
- **造价联动。** 根据模型自动生成工程量清单和概预算，打通"设计 -> 造价"的数据链路。
- **移动端远程控制。** 我有一个叫 RevitClaw 的构想——用手机远程发送建模指令到办公室的 Revit，随时随地"抓娃娃式建模"。目前这个市场方向还是空白的。

### Q19: 如果要商业化，你会怎么做？

**A:**

- **目标用户。** 两类：一是中小设计院（人少但项目多，需要提效工具），二是高校教学（降低 BIM 课程的入门门槛）。
- **定价模式。** Freemium——参数化框架生成免费，AI 对话功能按调用量收费（大模型 API 本身有成本）。
- **差异化。** 做中文市场最好用的 AI BIM 工具，竞品（主要是英文生态的）在中文建筑术语理解上有天然短板。
- **下一步。** RevitClaw 移动端是切入点——设计师出差在外想改模型，掏出手机发条语音就行。这个场景痛点明确，技术上用 WebSocket + 现有引擎就能实现。

> **加分回答：** "商业化不是只写代码就行。需要关注的还有：用户数据安全（建模指令可能包含商业机密）、API 成本控制（批量请求需要缓存和限流）、合规性（建筑设计软件可能涉及资质要求）。"

## 六、挑战性问题

### Q20: 如果大模型 API 不可用怎么办？

**A:**

- **参数化模式完全不受影响。** 一键生成框架、Excel 导入、构件修改/删除这些功能都是本地执行的，不依赖任何网络请求。
- **AI 模式有多层保障。** API 调用失败时自动重试（指数退避，最多 2 次）；5xx 错误和网络超时触发重试，非暂态错误直接报错并给出用户友好的恢复建议。
- **可切换提供商。** 配置文件里改一下 API 地址和密钥就能切换到其他兼容 OpenAI 格式的服务（包括本地部署的模型）。
- **指令回放。** 用户可以用 `/replay`、`/replaylog`、`/replayfail` 回放历史指令，不需要重新调用 API。

### Q21: 这个项目的局限性是什么？

**A:**

- **结构体系有限。** 目前只支持框架结构，剪力墙、钢结构、空间网架等体系还没有覆盖。这是工作量问题，架构上已经预留了扩展接口。
- **依赖网络。** AI 对话功能需要网络访问 DeepSeek API。虽然有重试和降级机制，但在完全断网的工地环境下，AI 模式不可用。
- **单语言。** 目前只支持中文输入。虽然 prompt 和别名系统理论上可以扩展到英文，但还没有做。
- **单用户。** 不支持多人协同建模。当前设计是单机 Revit 插件，没有并发控制。
- **精度依赖模型。** AI 对复杂指令（比如"参考 A 栋的结构在 B 地块上做一个类似的"）的理解能力有限，需要用户给出具体参数。

> **加分回答：** "谈局限性不是示弱，是展示对项目边界的清醒认知。我知道哪些问题是当前版本有意不解决的，哪些是后续版本的重点。"

### Q22: 你在这个项目中学到了什么？

**A:**

- **技术层面：**
  - Revit API 的体系和 BIM 数据模型（Level、FamilyInstance、Transaction 等概念）。
  - Prompt Engineering——怎么约束大模型输出稳定可靠的结构化数据，而不是自由发挥。
  - 测试驱动开发（TDD）——289 个测试不是事后补的，是边开发边写的，它们定义了每个函数的行为契约。
  - 跨平台兼容——IronPython vs CPython 的差异、.NET HTTP vs Python requests 的取舍。
- **工程层面：**
  - 架构设计的重要性——引擎层解耦这个决策在项目前期就做了，后面省了大量维护成本。
  - 配置管理——三级优先级的配置体系让部署变得灵活。
  - 错误处理不是可选项——用户体验的好坏很大程度上取决于出错时系统怎么应对。
- **认知层面：**
  - AI 不是万能的，但在合理约束下可以成为强大的工程工具。关键是把它放在一个可控的框架里，而不是让它自由发挥。
  - 好的软件不是功能最多的，而是每个功能都经过测试、有容错、有文档的。
